## Context

C4Flow is a pure markdown + shell skills plugin with no build step. The state machine is persisted at `docs/c4flow/.state.json`. Two skills — `infra` and `deploy` — are currently stubs. The `DEPLOY` state exists in the orchestrator but `INFRA` does not; both need to be wired in.

External tools already used by the plugin: `gh` (PR skill), `terraform` (init skill GitHub bootstrap), `bd` (task management). The infra and deploy skills extend these existing dependencies.

## Goals / Non-Goals

**Goals:**
- Implement `c4flow:infra` — EC2 + nginx provisioning via Terraform, Cloudflare DNS subdomain, GitHub Secrets push
- Implement `c4flow:deploy` — GitHub Actions SSH deploy workflow + nginx reverse proxy config generation
- Add `INFRA` as an explicit orchestrator state between `CODE_LOOP` and `DEPLOY`
- Config resolution at INFRA phase: env vars (detected, never displayed) → `.state.json` → interactive prompt

**Non-Goals:**
- Multiple deployment targets (ECS, Lambda, App Runner) — EC2 + nginx only
- Terraform remote state backend — local state only (user can migrate manually)
- Init phase changes — no infra questions added to `c4flow:init`
- SSL certificate provisioning — Cloudflare proxied records handle TLS at the edge
- Database or cache provisioning — app-level concerns, not handled here

## Decisions

### Decision 1: INFRA as explicit state (not hidden inside DEPLOY)

**Chosen**: `CODE_LOOP → INFRA → DEPLOY → DONE` — INFRA is a visible state in the orchestrator.

**Alternatives considered**:
- DEPLOY auto-triggers INFRA as its Step 1 — hides infra from `/c4flow:status`, harder to resume if infra fails mid-run.

**Rationale**: Explicit state means the user can see where they are, re-run `/c4flow:infra` standalone, and the orchestrator can handle failures at the right granularity.

### Decision 2: Config resolution in INFRA phase, not init

**Chosen**: INFRA's Step 1 resolves `domain`, `subdomain`, `awsRegion` at runtime.

**Alternatives considered**:
- Ask during `c4flow:init` — too early, couples project tooling setup with deployment concerns.

**Rationale**: Infra config only matters when infra runs. Deferring keeps init focused and avoids config questions on projects that never deploy.

### Decision 3: Env var detection without value display

When checking env vars, the skill MUST only acknowledge presence (`[configured via env]`), never print the value. Applies to: `C4FLOW_DOMAIN`, `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, `CLOUDFLARE_API_TOKEN`, `CLOUDFLARE_ZONE_ID`.

Non-sensitive config (`C4FLOW_AWS_REGION`, `C4FLOW_SUBDOMAIN`) may be displayed.

### Decision 4: Subdomain default

Default subdomain = `basename $(pwd)`. Stored in `infraConfig.subdomain`. Full FQDN computed as `<subdomain>.<domain>` and stored in `infraState.fqdn` after apply.

### Decision 5: Terraform — local state, inline HCL generation

INFRA generates Terraform HCL files into a temp working dir (`docs/c4flow/terraform/<feature-slug>/`), runs `terraform init` and `terraform apply -auto-approve`. Output is read via `terraform output -json`.

Local state file stays in that directory. User can migrate to S3 backend manually.

### Decision 6: GitHub Secrets pushed after apply

After terraform apply, INFRA runs `gh secret set` for exactly four secrets:
```
AWS_EC2_HOST        → EC2 public IP from terraform output
AWS_REGION          → awsRegion from infraConfig
EC2_SSH_PRIVATE_KEY → private key generated by Terraform (tls_private_key resource)
DEPLOY_DOMAIN       → fqdn from infraState
```

### Decision 7: nginx + SSL belong to INFRA, DEPLOY is CI/CD only

INFRA's `user_data` script installs nginx, runs certbot for Let's Encrypt SSL, configures the reverse proxy to `localhost:<appPort>`, and sets up certbot auto-renewal. This is infrastructure state — it lives on the EC2 instance permanently.

DEPLOY only generates `.github/workflows/deploy.yml` (SSH deploy + health check), commits and pushes to main. The first deployment runs via the GitHub Actions trigger. DEPLOY polls status via `gh run watch`.

## Risks / Trade-offs

- **Terraform apply is destructive and slow** → INFRA shows a plan summary and asks for confirmation before `apply`. Re-runs check if `infraState` already exists in `.state.json` and skip apply if infra is up.
- **Local terraform state lost if directory deleted** → Documented limitation. User is warned in INFRA output to not delete `docs/c4flow/terraform/`.
- **SSH key in GitHub Secrets** → Generated by Terraform `tls_private_key`, stored only in terraform state and GitHub Secrets. Never written to `.state.json`.
- **Cloudflare Zone ID required** → User must know their zone ID. INFRA prompts for it if `CLOUDFLARE_ZONE_ID` env var is absent; writes non-sensitive zone context to `infraConfig` if user provides it explicitly (but never the API token).
- **EC2 nginx serves all traffic** → No ALB, no auto-scaling. Appropriate for the single-app use case this skill targets.
- **certbot requires DNS to resolve before certificate issuance** → Cloudflare DNS propagation must complete before certbot runs. INFRA's `user_data` script adds a 60s sleep after instance start before running certbot, with retry logic. [Risk] DNS not propagated → certbot fails silently in user_data; INFRA skill verifies SSL by curling `https://<fqdn>` after apply and warns if cert is not yet valid.
- **App port must be known at Terraform generation time** → nginx config is baked into `user_data`. If the app port changes after infra is provisioned, the user must re-run `/c4flow:infra`. This is documented in INFRA output.
