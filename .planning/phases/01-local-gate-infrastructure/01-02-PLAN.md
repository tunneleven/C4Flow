Replace the stub `skills/review/SKILL.md` with a complete skill implementation. The skill is a markdown file that Claude reads and executes as instructions. Follow the established pattern from `skills/beads/SKILL.md` and `skills/c4flow/SKILL.md`.

**Frontmatter:**
```yaml
---
name: c4flow:review
description: AI code review via Codex subagent — writes quality-gate-status.json, resolves beads gate on pass
---
```

**Skill body must implement these steps IN ORDER:**

**Step 1: Tool availability detection (GATE-04)**
- Check `command -v bd` and `command -v codex`
- If bd missing: warn user, explain that beads gate management is unavailable, offer manual review checklist as fallback, exit gracefully
- If codex missing: warn user, explain that AI review is unavailable. If bd IS available, create a manual gate: `bd create "Manual review required (codex not available)" --labels "c4flow-quality-gate,gate,manual"`. Print manual review instructions. Exit gracefully
- If both available: proceed to Step 2

**Step 2: Check for existing gate status and expiry**
- Read `quality-gate-status.json` if it exists
- If file exists AND `expires_at` is in the future: inform user that a valid (non-expired) review exists. Show the summary. Ask: "Re-run review or use existing results?" If user chooses existing, skip to Step 5
- If file exists but expired: inform user that results are stale. Proceed to re-run
- If file does not exist: proceed to Step 3

**Step 3: Gate creation / lookup (GATE-03)**
- Read `quality-gate-status.json` for existing `gate_id` field (pitfall 3: prevents lost gate ID between invocations)
- If `gate_id` exists and is non-null: verify gate still exists via `bd gate list --json | jq -r '.[] | select(.id == "<gate_id>") | .id'`. If found, reuse it
- If no existing gate_id or gate not found: create new gate:
  ```bash
  GATE_ID=$(bd create "Quality gate: c4flow review+verify" \
    --labels "c4flow-quality-gate" \
    --description "Automated gate: resolves when Codex review AND bd preflight pass" \
    --json | jq -r '.id')
  ```
- Store GATE_ID for use in quality-gate-status.json (written in Step 4)
- Label-based fallback lookup if creation fails: `bd gate list --json | jq -r '.[] | select(.labels[] | contains("c4flow-quality-gate")) | .id' | head -1`

**Step 4: Dispatch code-reviewer subagent and write results (GATE-01, SKIL-01)**
- Dispatch the code-reviewer subagent (`.claude/agents/code-reviewer.md`)
- The subagent runs Codex and returns structured JSON
- Parse the subagent output: extract JSON using `echo "$OUTPUT" | grep -o '{.*}' | jq .` (pitfall 1: handle prose wrapping)
- Validate required fields: `jq 'has("pass") and has("critical_count") and has("findings")'`
- If parse fails: set `codex_review.pass = false`, summary = "Failed to parse subagent output", findings = []
- Write to `quality-gate-status.json` using atomic write pattern (pitfall 2):
  1. Read existing file to preserve bd_preflight results if present
  2. Merge codex_review results into the checks object
  3. Write to temp file first, then `mv` to final path
  4. Set `generated_at` to current time, `expires_at` to generated_at + 60 minutes (configurable via `C4FLOW_GATE_EXPIRY_MINUTES` env var)
  5. Set `gate_id` to the gate ID from Step 3
  6. Set `overall_pass` = true ONLY if codex_review.pass == true AND bd_preflight.pass == true. If bd_preflight.pass is null, overall_pass MUST be false

**Step 5: Gate resolution or report (INFR-04)**
- If `codex_review.pass == true`:
  - Check if `bd_preflight.pass == true` as well (from existing file)
  - If BOTH pass: resolve gate with audit trail:
    ```bash
    bd gate resolve "$GATE_ID" --reason "Codex review: 0 critical, 0 high (M:$MEDIUM_COUNT, L:$LOW_COUNT). bd preflight: passed. Reviewed at: $TIMESTAMP"
    ```
  - If only codex passes but preflight not yet run: report "Codex review passed. Run c4flow:verify to complete preflight check."
- If `codex_review.pass == false`:
  - Report findings to user in a readable format: group by severity, show file:line and message
  - Remind user: "Fix CRITICAL and HIGH issues, then re-run `/c4flow:review`"
  - Gate remains open (no bd gate resolve call)

**Step 6: Summary output**
- Print a summary box showing: overall status, per-check status, finding counts by severity, gate ID, expiry time
- If review passed: "Review passed. Quality gate status written to quality-gate-status.json"
- If review failed: "Review failed. N critical, M high findings must be resolved."

**Important constraints (from locked decisions):**
- Report and stop pattern — NO auto-fix loop. Skill reports findings and exits. User fixes manually and re-runs
- CRITICAL + HIGH block only. MEDIUM/LOW are informational
- Branch diff vs main: the subagent uses `codex review --base main`
- Single combined gate per task (not separate gates for review vs preflight)
