<task type="auto">
  <name>Task 1: Define quality-gate-status.json schema and add .gitignore entry</name>
  <files>quality-gate-status.schema.json, .gitignore</files>
  <action>
Create `quality-gate-status.schema.json` as a JSON Schema (draft-07) that formally defines the structure of `quality-gate-status.json`. The schema must include:

**Root fields:**
- `schema_version` (string, enum: ["1"]) — versioning for future changes
- `generated_at` (string, date-time format) — when file was last written
- `expires_at` (string, date-time format) — when results become stale (default: generated_at + 60 minutes)
- `gate_id` (string or null) — beads gate ID stored at creation time (per pitfall 3: prevents lost gate ID between invocations)
- `overall_pass` (boolean) — true ONLY when ALL checks have pass=true (never true when any check is null/not-yet-run, per anti-pattern in RESEARCH.md)
- `checks` (object) — container for per-check results

**checks.codex_review fields:**
- `pass` (boolean or null) — null means not yet run
- `ran_at` (string date-time or null)
- `critical_count`, `high_count`, `medium_count`, `low_count` (integer, minimum 0)
- `findings` (array of finding objects)
- Each finding: `severity` (enum: CRITICAL, HIGH, MEDIUM, LOW), `file` (string), `line` (integer or null), `message` (string)

**checks.bd_preflight fields:**
- `pass` (boolean or null)
- `ran_at` (string date-time or null)
- `issues` (array of objects with `check` string and `message` string)

Mark `schema_version`, `generated_at`, `expires_at`, `overall_pass`, `checks` as required at root. Mark `pass`, `ran_at` as required within each check object.

Then update `.gitignore` to add `quality-gate-status.json` on its own line. Read the existing `.gitignore` first — it currently has `.dolt/` and `*.db`. Append below existing entries with a blank line and comment: `# C4Flow quality gate (ephemeral, regenerated each review run)`.
  </action>

<task type="auto">
  <name>Task 2: Create project-scoped code-reviewer subagent definition</name>
  <files>.claude/agents/code-reviewer.md</files>
  <action>
Create `.claude/agents/` directory if it does not exist, then create `.claude/agents/code-reviewer.md`.

This is a project-scoped override of the global `~/.claude/agents/code-reviewer.md`. It must be tailored for C4Flow quality gate output.

**Frontmatter (YAML):**
```yaml
---
name: code-reviewer
description: C4Flow code review subagent — runs Codex review and returns structured JSON for quality gate evaluation
tools: ["Bash", "Read"]
model: sonnet
---
```

**Body — the subagent prompt:**

1. **Identity:** "You are a code review subagent for the C4Flow quality gate chain."

2. **Tool availability check:** Before running Codex, check `command -v codex`. If not found:
   - Output a JSON object with `pass: false`, all counts at 0, empty findings array, and summary: "Codex CLI not found. Manual review required."
   - This satisfies GATE-04 (graceful fallback when codex missing) at the subagent level.

3. **Codex invocation:** Run `timeout 120 codex review --base main` synchronously. Do NOT background it (pitfall 6). Capture full stdout+stderr.

4. **Output parsing instructions:** Parse the Codex prose output and classify each finding by severity (CRITICAL, HIGH, MEDIUM, LOW). The subagent must use judgment to classify — Codex does not output structured severity levels natively.

5. **Pass/fail logic (per locked decision):** `pass: true` if and only if `critical_count == 0 AND high_count == 0`. MEDIUM and LOW findings are informational only — they do NOT block gate resolution.

6. **Output contract — the ONLY output must be a JSON object:**
```json
{
  "pass": <boolean>,
  "critical_count": <integer>,
  "high_count": <integer>,
  "medium_count": <integer>,
  "low_count": <integer>,
  "findings": [
    {"severity": "CRITICAL|HIGH|MEDIUM|LOW", "file": "<path>", "line": <integer or null>, "message": "<description>"}
  ],
  "summary": "<one sentence summary>"
}
```

7. **Strict output rule:** "Do NOT output any text before or after the JSON object. No markdown fences. No explanatory prose. The calling skill extracts JSON from your output — any non-JSON text will cause a parse failure and the gate will remain open (fail-safe)."

8. **Error handling:** If `codex review` times out or returns an error, output: `{"pass": false, "critical_count": 0, "high_count": 0, "medium_count": 0, "low_count": 0, "findings": [], "summary": "Codex review failed: <error description>. Manual review required."}`
  </action>

  </verify>
  <done>code-reviewer.md exists at .claude/agents/ with: YAML frontmatter (name, tools, model), Codex invocation with 120s timeout, tool availability check for codex, structured JSON output contract matching the codex_review section of quality-gate-status.schema.json, fail-safe error handling.</done>
