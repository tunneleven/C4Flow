  <action>
Replace the stub at skills/pr/SKILL.md with the full skill implementation. Follow the existing c4flow:verify SKILL.md pattern (step-by-step markdown instructions).

The skill MUST have these steps in order:

**Step 1: Tool Availability Detection**
- Check `command -v gh`. If missing: print manual PR checklist (push branch, open GitHub compare URL, include gate summary manually, update .state.json manually) and exit gracefully (exit 0, not exit 1).
- Check `gh auth status`. If not authenticated: warn and exit 1.

**Step 2: Read Gate Status and Warn if Not Passed**
- Read quality-gate-status.json with jq. Check overall_pass.
- If file missing: warn "No quality-gate-status.json found. Run /c4flow:review and /c4flow:verify first." Set a flag for fallback body.
- If overall_pass is not true: print WARNING with current status, ask user "Proceed anyway? (yes/no)". If no: exit. If yes: continue.
- If overall_pass is true: proceed silently.
- CRITICAL: This is a WARN, not a BLOCK. Never exit 1 solely because gates failed — always offer the user a choice.

**Step 3: Build PR Body**
- Construct a markdown PR body with a "## Quality Gate Status" section containing a table.
- Table columns: Check | Status | Details | Ran At
- Read each check from quality-gate-status.json using jq (codex_review and bd_preflight).
- Use emoji indicators: checkmark PASS, X FAIL, warning NOT RUN.
- Include overall status line and gate_id/timestamps at bottom.
- Handle missing file case: use "No status available" fallback body.
- Write the body to a temp file to avoid shell escaping issues (Pitfall 3 from research).
- Use `--body-file` with the temp file, not inline `--body`.

**Step 4: Determine PR Title**
- Read docs/c4flow/.state.json for feature.name if it exists.
- Default title: "feat: {feature.name}" if available, otherwise use current branch name cleaned up.
- Print the proposed title and let user confirm or customize.

**Step 5: Check for Existing PR**
- Run `gh pr view --json number --jq '.number' 2>/dev/null`.
- If a PR already exists: print its number and URL, update .state.json if prNumber is null, and exit 0. Do NOT create a duplicate (Pitfall 2 from research).

**Step 6: Push Branch if Not Pushed**
- Get branch name: `git rev-parse --abbrev-ref HEAD`.
- Check remote: `git ls-remote --heads origin "$BRANCH"`.
- If not pushed: run `git push -u origin "$BRANCH"`.

**Step 7: Create PR**
- Run `gh pr create --title "$PR_TITLE" --body-file "$BODY_FILE" --base main`.
- Capture the output URL.
- Extract PR number: `echo "$PR_URL" | grep -o '[0-9]*$'`.
- Clean up the temp body file.

**Step 8: Write PR Number to .state.json**
- Use the atomic jq merge pattern (NOT overwrite): `jq --argjson num "$PR_NUMBER" '.prNumber = $num | .currentState = "PR_REVIEW_LOOP" | .completedStates += ["PR"] | .failedAttempts = 0 | .lastError = null' "$STATE_FILE" > "${STATE_FILE}.tmp" && mv "${STATE_FILE}.tmp" "$STATE_FILE"`.
- If .state.json does not exist: create minimal with prNumber only and warn.

**Step 9: Summary Output**
- Print: PR URL, PR number, gate status (passed/not passed), next step hint ("PR is open. Reviewers will see the gate summary. Run /c4flow:pr-review when ready to check review status.").

Frontmatter must be:
```yaml
---
name: c4flow:pr
description: Create a GitHub PR with quality gate status summary in description, write PR number to .state.json
---
```

Keep each step as a ### heading with bash code blocks showing exact commands. Total file should be 150-250 lines.
